# Ultron — local voice-first AI assistant prototype

A continuously-listening voice assistant that runs entirely on your hardware.
Says "ultron" → captures your request → transcribes → asks a local LLM →
speaks the response. No cloud round-trips, no telemetry.

## Setup

### 1. Verify CUDA

```powershell
nvidia-smi
```

You want CUDA 12.x or 13.x and a working driver. If `nvidia-smi` errors,
fix that first — nothing else will work.

### 2. Clone & create a virtual environment

```powershell
git clone https://github.com/1v9Khan/ultronPrototype.git ultronPrototype
cd ultronPrototype
python -m venv .venv
.venv\Scripts\activate
pip install -U pip wheel
```

### 3. Install dependencies

```powershell
pip install -e .
```

Then install `llama-cpp-python` **with CUDA support** (the wheels on
PyPI default to CPU-only). Pick the wheel matching your CUDA toolkit:

```powershell
# CUDA 13.x driver (recommended):
pip install llama-cpp-python --extra-index-url https://abetlen.github.io/llama-cpp-python/whl/cu130

# CUDA 12.x driver:
pip install llama-cpp-python --extra-index-url https://abetlen.github.io/llama-cpp-python/whl/cu124
```

Run `nvidia-smi` — the "CUDA Version" in the top-right is your
driver's max supported toolkit. Match the wheel suffix to that. Check
[the wheel index](https://abetlen.github.io/llama-cpp-python/whl/) for
other versions.

### CUDA 13 compatibility note

CTranslate2 (faster-whisper) and onnxruntime-gpu (openWakeWord) don't
yet ship CUDA 13 wheels. Install them as normal from PyPI — their
CUDA 12 builds run fine on a CUDA 13 driver via NVIDIA's forward
compatibility. No changes needed.

### 4. Download models

```powershell
python scripts/download_models.py
```

This pulls (~9 GB total):

- Qwen3.5-9B Q4_K_M GGUF (LLM, ~5.7 GB) — kept for swap-back
- Qwen3.5-4B Q4_K_M GGUF (~2.7 GB) — active default
- Qwen3.5-0.8B Q4_K_M GGUF (~530 MB) — speculative-decoding draft for 4B
- Piper voice `en_US-ryan-medium` (TTS)
- faster-whisper `small.en` (STT)
- openWakeWord pretrained models (`hey_jarvis` etc.)

The custom **Ultron** wake-word ONNX is **not** downloaded — see
[Wake word](#wake-word) below.

### 5. Run

```powershell
python -m ultron
```

First run takes 1–3 minutes as models warm up. Subsequent starts are
faster because the OS-level page cache holds the GGUF.

Say the wake word, then speak. Press **Ctrl+C** to shut down.

## Wake word

The user-facing wake word is **"ultron"**. openWakeWord ships pretrained
models for `alexa`, `hey_jarvis`, `hey_mycroft`, `hey_rhasspy`, and
`weather` — **not** Ultron.

A custom-trained `models/openwakeword/ultron.onnx` is the production
wake word. Out of the box (without that file), the prototype falls back
to **`hey_jarvis`** with a loud warning at startup. To get true
`ultron` detection:

1. Train a custom model — see openWakeWord's
   [automatic training notebook](https://github.com/dscripka/openWakeWord/blob/main/notebooks/automatic_model_training.ipynb).
   The notebook generates synthetic samples of the target word and trains a
   small classifier on top of pretrained embeddings; runtime is ~1 hour on a
   Colab T4.
2. Place the resulting `ultron.onnx` at `models/openwakeword/ultron.onnx`
   (or override `wake_word.model_path` in `config.yaml`).
3. Restart `python -m ultron`. The startup banner will confirm the
   custom model loaded instead of the fallback.

## Architecture

```
mic → wake word ("ultron") OR addressing classifier (WARM mode)
    → VAD-bounded utterance capture (mode-aware ring-buffer pre-roll: COLD 0.15 s / WARM 0.5 s)
    → Whisper STT
    → classify_routing() ── coding ── CodingTaskRunner (Claude Code subprocess)
                         ├ conversational ── LLM (Qwen3.5-4B Q4 via llama-cpp-python)
                         │                   ├─ optional pre-flight web-search gate
                         │                   │  ├─ Brave + Jina (real, parallel-fetch)
                         │                   │  └─ acknowledgment phrase to TTS in <200 ms
                         │                   ├─ prompt-injection defense layer (Q10)
                         │                   ├─ brevity hint for short conversational turns
                         │                   └─ stream tokens to TTS engine
                         │                       ├ tts.engine: "xtts_v3"  → XTTS v2 streaming → v3 pedalboard filter → audio (default, ~140 ms TTFT)
                         │                       └ tts.engine: "piper_rvc" → Piper → RVC → audio (legacy fallback, 79 ms TTFT)
                         ├ openclaw ── live dispatch via OpenClaw bridge (CLI subprocess)
                         │             ├ messaging (Telegram channel)
                         │             ├ browser (Playwright)
                         │             ├ media generation (ComfyUI, local-only)
                         │             ├ desktop / window control (V1-gap C3)
                         │             ├ gaming mode (V1-gap A1)
                         │             └ system status (Phase 13 SYSTEM_STATUS intent)
                         └ hybrid ── HybridTaskDecomposer split into subtasks
    → async write turn to Qdrant (memory)
    → enter WARM mode (30 s follow-up window)
```

### Design rules (binding)

- **Voice-quality lock (partial lift, 2026-05-10):** Piper, RVC, and the
  LLM model file remain locked — voice character on the legacy
  `piper_rvc` stack is non-negotiable. The default `xtts_v3` engine
  (`src/ultron/tts/xtts_v3.py` + `ultron_filter.py`) is the appropriate
  place to iterate on voice character now.
- **Voice baseline:** TTFT median ≤ 79 ms, VRAM peak ≤ 7913 MB on 4B
  preset. Re-measure via `python scripts/measure_baseline.py` after
  any change touching the hot path.
- **No paid APIs except Claude Code.** Brave Search and Jina Reader use
  free tiers; ComfyUI is local; Telegram bots are free. Fal / Runway /
  Suno / OpenAI image-gen / Azure / ElevenLabs are explicitly out.
- **Fail-open everywhere.** External-dependency failures must NEVER
  crash the orchestrator — log WARN, degrade gracefully, return a
  voice message in Ultron's character.
- **`docs/codebase_structure.md` updated alongside any code change**
  per its binding maintenance contract.
- **`tools.profile: "messaging"`** locked on every local-Qwen OpenClaw
  agent (the `coding` profile bundles 50+ tool schemas that exceed
  Qwen3.5-9B prefill in the hardcoded 30 s OpenAI SDK timeout).

## Customizing

### Voice

Two TTS engines ship; pick via `tts.engine` in `config.yaml`.

**`xtts_v3` (default, ~140 ms TTFT):** XTTS v2 zero-shot voice cloning
+ a v3 pedalboard filter chain for Ultron mechanical character.
Provide a clean ~10 s reference clip at `tts.xtts_v3.reference_audio`.
Server runs in an isolated venv (`.venv-xtts`) — see
[`ultronVoiceAudio/README.md`](ultronVoiceAudio/README.md) for the
cold-start setup. Filter preset defaults to `v3_heavy` (the
production-locked one). Costs ~2 GB extra VRAM for the XTTS model.

**`piper_rvc` (legacy fallback, 79 ms TTFT):** Set
`tts.engine: "piper_rvc"` to use the lower-latency Piper + RVC timbre
stack. Browse [Piper voices](https://huggingface.co/rhasspy/piper-voices/tree/main/en/en_US)
and put any `.onnx` + `.onnx.json` pair under `models/piper/`. Update
`tts.piper_voice_path` and `tts.piper_voice_config_path` in
`config.yaml`. For Ultron's character, low and slow voices (e.g.
`en_US-ryan-low`, `en_GB-alan-medium`) work better than bright ones.
The active RVC voice (Ultron timbre) lives at
`ultron_james_spader_mcu_6941/Ultron.pth`.

### LLM preset switching

The active preset is `qwen3.5-4b` (the `qwen3.5-9b` GGUF stays in
`models/` for swap-back). Switch via:

- **Voice command:** "Ultron, switch to the 9B" / "Ultron, use the 4B"
- **Helper script:** `python scripts/swap_llm_preset.py qwen3.5-9b`
- **Env var:** `$env:ULTRON_LLM_PRESET = "qwen3.5-9b"; python -m ultron`
- **YAML edit:** `llm.preset: "qwen3.5-9b"` (one-line edit)

The voice command swap takes ~1.7 s for 4B, ~3–5 s for 9B (in-process
build-then-swap; failure-safe).

### System prompt / persona

Persona lives in `~/.openclaw/workspace/{IDENTITY,SOUL,USER,AGENTS,
HEARTBEAT,BOOTSTRAP}.md`. Edits hot-reload on the next user turn —
no restart. **Don't modify SOUL.md without explicit consideration of
the voice-quality lock** (Stage E A/B verification was passed
2026-05-08 against the current state).

## Troubleshooting

**`nvidia-smi` works but llama-cpp says CPU-only.**
You installed the default wheel from PyPI. Reinstall with the CUDA
index matching your driver (cu130 for CUDA 13.x, cu124 for CUDA
12.x):
```powershell
pip uninstall llama-cpp-python
pip install llama-cpp-python --extra-index-url https://abetlen.github.io/llama-cpp-python/whl/cu130
```

**Whisper crashes with cuBLAS or cuDNN errors on Windows.**
faster-whisper (via CTranslate2) is built against **CUDA 12 / cuDNN 9**.
It works on a CUDA 13 driver via forward compat, but it still needs
cuDNN 9 DLLs on PATH — those don't ship with the CUDA 13 toolkit by
default. Install [cuDNN 9 for CUDA 12.x](https://developer.nvidia.com/cudnn)
and either add the install dir to PATH or copy `cublas64_12.dll` and
`cudnn_ops_infer64_9.dll` next to your venv.

**No audio device / mic not found.**
```powershell
python scripts/list_audio_devices.py
```
Add your device's name (or a unique substring) to `.env`:
```
ULTRON_AUDIO_DEVICE=Yeti
```

**LLM runs out of VRAM.**
The 4B preset is the active default and fits comfortably (peak ~7.9
GB). If running 9B preset on a smaller card, options:

- Voice command: "Ultron, switch to the 4B" — runtime swap, no restart
- Lower `llm.gpu_layers` (e.g. `30` instead of `-1`) to spill some
  layers to CPU
- Drop Whisper to `base.en` (frees ~200 MB)

**Wake word triggers on the assistant's own voice.**
Expected — there's no echo cancellation in the prototype. Mitigations:
move the speakers away from the mic, lower `llm.default_max_tokens`
to shorten responses, or raise `wake_word.threshold` toward `0.7`.

**First-token latency is way above 100 ms.**
Run `python scripts/measure_baseline.py` to see which stage is slow.
Common causes:

- Stale `.env` `ULTRON_LLM_MODEL_PATH` overriding the preset (comment
  it out)
- Whisper model too large for the GPU (drop to `base.en`)
- LLM not actually on GPU (check the install command above)
- Cold cache on first run (always slower; re-test after one warm
  round-trip)

**Can't reach OpenClaw.**
The bridge defaults to `openclaw.enabled: false` so this can't crash
voice. To enable the live integration:
1. Install OpenClaw Desktop / configure `~/.openclaw/openclaw.json`
2. Start `llama-cpp-server` on `127.0.0.1:8765` via
   `python scripts/start_llamacpp_server.py --from-config`
3. Start the OpenClaw Gateway
4. Set `openclaw.enabled: true` in `config.yaml`

See `docs/openclaw_runtime.md` for the full procedure.
