# Auto push-to-talk firmware (Arduino Leonardo)

Lets Ultron transmit relay callouts on Valorant **team** voice (which is
push-to-talk only). The PC host (`kenning.ptt`) writes one byte to a serial
port; this board presses a **real USB-HID key**. The host never performs
synthetic input — the keypress is generated by physical hardware with no
injected flag, which is the anticheat-clean design (see
`src/kenning/safety/anticheat.py`'s "Deliberately NOT blocked" list).

## Wire protocol (host → board)

| Byte | Meaning   | Board action                                   |
|------|-----------|------------------------------------------------|
| `D`  | key down  | `Keyboard.press(KEY)`                          |
| `U`  | key up    | `Keyboard.release(KEY)`                        |
| `H`  | heartbeat | keep holding; refresh the hardware deadman     |

**Hardware deadman:** if the board stops receiving bytes for `DEADMAN_MS`
(default 200 ms) while holding, it auto-releases the key — so a host crash or a
yanked USB cable can never jam your mic open mid-match.

## Slot-in steps (when you find the Leonardo)

1. In `leonardo_ptt.ino`, set `KEY` to the key you bind to **Team Voice** PTT in
   Valorant (Settings → Audio → Voice Chat). **This rig: `'6'`.**
2. Flash with the Arduino IDE — board: **Arduino Leonardo** — or with
   `arduino-cli compile/upload --fqbn arduino:avr:leonardo firmware/leonardo_ptt`
   (needs the `Keyboard` library: `arduino-cli lib install Keyboard`).
3. Note the COM port (Device Manager → Ports). It can **drift across flashes**,
   so config defaults to `"auto"` (detect by USB id) rather than a fixed port.
4. Arm it — either the settings panel, or `.env` on this machine:
   - `KENNING_PTT_ENABLED=true`
   - `KENNING_PTT_SERIAL_PORT=auto`  (or an explicit `COM5`)
   - keep `push_to_talk.key` (`"6"`) == the firmware `KEY` == your Valorant bind
5. Reboot Ultron. The boot log will show
   `push-to-talk: auto-detected Arduino on COMx` → `… ARMED via external USB-HID …`.

Until then it stays **inert** (a `NullPttBackend` that does nothing). If the
port is wrong or the device is unplugged, PTT fails **safe** — it simply doesn't
fire, and never falls back to in-process input.

## Tuning (config `push_to_talk.*`)

- `lead_ms` (120) — hold the key this long *before* audio so the first phoneme
  isn't clipped by the game opening the channel.
- `release_tail_ms` (150) — keep holding this long *after* the clip drains (the
  reverb tail is already baked into the clip; this covers the game codec tail).
- `heartbeat_ms` (50) — keep-alive cadence; keep it well under the firmware
  `DEADMAN_MS`.
- `max_hold_seconds` (8) — host-side watchdog; the firmware `MAX_HOLD_MS` (15 s)
  is the hardware backstop.
